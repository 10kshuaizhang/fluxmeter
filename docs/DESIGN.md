## Problem Statement

AI token billing has unique semantics that existing metering engines handle poorly: prompt vs completion tokens, per-model pricing tiers, streaming chunk aggregation, embedding dimension-based billing. Every existing solution (OpenMeter, Orb, Metronome) treats metering as an analytics problem (store events, query later). No open source project applies true stream processing to metering, despite streaming being architecturally superior for real-time aggregation at massive scale.

## Target Customer

**Primary:** AI application companies / SaaS platforms that consume tokens from LLM providers (OpenAI, Anthropic, etc.) and need to bill their own end users. Examples: Cursor-like code assistants, Perplexity-like search, vertical AI SaaS (legal, medical, customer service), AI agent platforms.

**Secondary:** AI infrastructure / platform companies providing LLM Gateway, model hosting, or agent orchestration that need to meter downstream developers.

**Not:** Top-tier model providers (OpenAI, Anthropic) -- they self-build.

**What these customers actually need (validated market pain):**

*Pillar A — Metering (shipped):*
1. Real-time guardrails: prevent runaway agents from burning budgets
2. Credits / prepaid drawdown: customers pre-purchase credits, deduct in real-time
3. Multi-provider normalization: OpenAI/Anthropic/Bedrock have different event schemas
4. Exactly-once + retroactive re-rating: no double-counting, recalculate on pricing changes
5. Streaming chunk handling: meter tokens as they stream, not just at completion

*Pillar B — Intelligence (Phase 5):*
6. Root cause: explain spend deltas by model, customer, feature, team
7. Unit economics: revenue vs cost, margin, loss alerts, simple recommendations
8. Scenario simulation: what-if for model switch, prompt optimization, promos
9. Overlay ingest: combine native meter data with Langfuse / OpenMeter / Helicone exports

**Primary audience (2026-07):** Founder, Finance, RevOps, Product — in addition to engineering/billing ops.

**Competitive positioning:** Layer 3 — open source + self-hosted + streaming-first + exactly-once. Layer 4 — prescriptive insights + simulation; complement observability and invoice platforms, do not replace them.

## What Makes This Cool

Three things converge:
1. **No one has done this.** Zero Flink-based open source metering engines exist. The entire space uses store-then-query.
2. **Streaming-first is provably better for this workload.** Sub-second aggregation latency, exactly-once via Flink checkpointing, natural backpressure handling. ClickHouse-based approaches hit 5-30 second query latency at scale.
3. **The demo is visceral.** A live Grafana dashboard showing 1M+ token events/sec streaming through Flink, side-by-side with a naive ClickHouse baseline choking on the same workload. Numbers that make people stop scrolling.

Future "whoa" factor (v2): Real-time budget enforcement that kills runaway LLM calls mid-stream when cost thresholds are crossed. Architecturally impossible without streaming-first design.

## Architecture: Java Core + Python SDK

```
[AI App] --Python SDK (pip install fluxmeter)--> [FastAPI Gateway] --Kafka--> [Flink Job (Java)] ---> [Redis/Grafana]
                                                                                                  \--> [ClickHouse (historical)]
```

- **Core engine:** Java 17 + Flink 1.18 DataStream API (maximum performance, deep Flink learning)
- **User-facing SDK:** Python (`fluxmeter-client` package) -- AI developers interact via Python
- **Query API:** Python FastAPI for usage queries, dashboard data
- **Why hybrid:** Java core hits 1M+ eps (no PyFlink serialization overhead). Python SDK ensures AI community adoption. The engine is Java; the interface is Python.

## Constraints

- Java 17 core engine (Flink's native language, maximum performance)
- Python SDK and API layer (AI community adoption)
- Must run on commodity hardware (docker-compose, 32GB RAM recommended for full benchmark)
- Must produce a shareable demo (GIF/video for HN/Twitter)
- Must teach the builder Flink deeply (DataStream API, not Flink SQL abstractions)
- Open source from day one (license TBD -- see Open Questions)

## Premises

1. AI token billing has unique semantics (prompt/completion tokens, model-specific pricing, streaming chunks) that justify a purpose-built engine rather than configuring a general metering tool like OpenMeter.
2. A streaming-first architecture (Flink) is meaningfully better than store-then-query (ClickHouse) for metering at scale -- sub-second latency and exactly-once matter to this audience.
3. A Java core + Python SDK hybrid can attract both audiences: billing engineers respect the Java/Flink depth, AI developers adopt via the Python SDK. The engine language doesn't matter if the interface is Python.
4. The "performance benchmark + real-time dashboard" demo is the right lead -- it proves architectural superiority over the naive approach and is visually impressive.
5. Building this publicly (open source, blog posts, benchmarks) will create career opportunities at companies like Orb, Metronome, or advance position at Zuora.

## Approaches Considered

### Approach A: Weekend Rocket (Performance-first demo)
Docker-compose with Flink + Kafka + Grafana. Load generator fires 1M+ synthetic token events/sec. Dashboard shows real-time aggregation vs ClickHouse baseline. Ship in one weekend.

- Effort: S (1 weekend)
- Risk: Low
- Pros: Ships fastest, visually impressive, forces deep Flink learning
- Cons: No real billing semantics (flat pricing only), might be dismissed as "just a benchmark"

### Approach B: Domain-first (AI billing semantics + moderate scale)
Build the token billing data model first (prompt/completion tokens, model pricing tiers, streaming chunk reassembly). Include a REST API for ingestion. Takes 3-4 weekends.

- Effort: M (3-4 weekends)
- Risk: Medium (scope creep)
- Pros: Proves domain understanding, usable by others, differentiates from "just a demo"
- Cons: Slower to ship, risk of billing edge case rabbit holes

### Approach C: Budget Enforcer (Real-time kill signal demo)
Lead with real-time budget enforcement. Flink maintains running cost per customer, emits kill signals when thresholds crossed. Demo shows LLM response terminated mid-stream at budget limit.

- Effort: M (2-3 weekends)
- Risk: Medium-High (needs mock inference proxy)
- Pros: Most memorable demo, architecturally impossible without streaming, solves real pain
- Cons: Narrower scope, more complex first Flink project

## Recommended Approach

**Approach A: Weekend Rocket.** Ship the benchmark demo in one weekend. The goal is a working artifact you can post on HN Monday morning. The domain depth (B) and killer feature (C) layer on top in subsequent weekends.

Sequence: A (weekend 1) -> Python SDK (weekend 2) -> C (weekends 3-4) -> B (ongoing iteration)

## Open Questions

1. **Baseline comparison:** ClickHouse Kafka engine with materialized views (polling every 5s), or simpler Kafka -> Postgres batch insert -> scheduled query? ClickHouse is a fairer comparison; Postgres makes the gap more dramatic.
2. Should the load generator model real OpenAI API response shapes, or simplified synthetic events?
3. GitHub org name: `fluxmeter` or personal account?
4. License: Apache 2.0 (max adoption) or AGPL (prevents cloud providers from hosting without contributing)?

**Resolved:**
- Kafka: KRaft-mode Apache Kafka (decided in Next Steps)
- Hardware: 32GB RAM recommended. For full benchmark, run Flink pipeline and ClickHouse baseline sequentially (not simultaneously) to avoid I/O contention. Docker memory limits per container: Kafka 4GB, Flink TMs 8GB each, ClickHouse 8GB, Redis 1GB.

## Deferred / Out of Scope (Weekend 1)

- Error handling for Flink checkpoint failures, Kafka broker loss, Redis downtime
- Late event handling and retry semantics
- Authentication and multi-tenancy
- Tests (unit or integration)
- Kubernetes deployment
- Complex pricing logic (tiered, committed-use discounts)
- "Pricing algebra engine" (mentioned in cross-model perspective) -- deferred to Approach B phase

## Success Criteria

**Build success (hard criteria):**
- At least 500K events/sec sustained on 32GB single machine; target 1M+ with documented linear scaling evidence
- Sub-second aggregation latency: p99 < 500ms from Kafka ingestion to Redis write. Measured by embedding `ingestion_ts` in event, computing delta at Flink window close, exporting histogram to Prometheus/Grafana.
- A 90-second terminal demo GIF that tells the whole story
- Clean README that explains what this is, why it exists, and how to run it in 30 seconds

**Launch success (aspirational):**
- HN post that gets 50+ upvotes (proves the concept resonates)
- 100+ GitHub stars in first week

## Next Steps

**Weekend 1 (Approach A):**
1. Init repo: Java 17, Gradle, Flink 1.18 dependencies
2. `docker-compose.yml`: Kafka (KRaft), Flink JobManager + 2 TaskManagers, Redis, Grafana
3. `TokenUsageAggregator` Flink job: keyed by (customer_id, model_id), 1-min tumbling window, running token counts + cost
4. Load generator: Java Kafka producer with batching, configurable parallelism, realistic token event shapes
5. Sink window-aggregated results to Redis (post-aggregation, not per-event). With 10K customers and 1-min windows, Redis sees ~167 writes/sec -- well within single-instance capacity. Use Flink Redis sink with pipelining.
6. Grafana dashboard: throughput, latency, per-customer cost, Flink vs baseline comparison
7. Naive baseline: ClickHouse Kafka engine consuming same topic, materialized view aggregating per (customer_id, model_id, 1-min window). Query polls every 5s. Table DDL and MV definition in `baseline/` directory.
8. `make demo` one-command startup
9. Record terminal GIF, write README
10. Post to HN: "FluxMeter: Stream-processing 1M token billing events/sec with Apache Flink"

**Weekend 2 (Python SDK + API):**
11. `fluxmeter-client` Python package: event emitter (Kafka producer wrapper), async/sync modes
12. FastAPI query endpoint: `GET /usage/customer/{id}`, `GET /usage/customer/{id}/period/{YYYY-MM}`, `GET /usage/customer/{id}/day/{YYYY-MM-DD}`, `GET /usage/session/{id}`, `GET /usage/span/{id}`, budget CRUD
13. Python examples in README: `pip install fluxmeter && fluxmeter.emit(event)` -- 3 lines to start metering
14. Load generator rewritten in Python (shows the SDK works end-to-end)

**Weekend 3-4 (Approach C -- Budget Enforcer):**
15. Add per-customer budget state in Flink keyed state
16. Budget enforcement: emit kill signals to a "control" Kafka topic when thresholds crossed
17. Mock inference proxy that streams tokens and respects kill signals
18. Demo: show a streaming response terminated mid-flight at $0.50 budget
